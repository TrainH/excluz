<br>
1.한 줄 요약

- 비즈니스 목표(4000) 및 과부하 상황(10000)에서 안정적 처리시간(25초, 40초 이내)을 보이는 낙관락 채택

-서버 사양
- 로컬 환경:CPU: AMD Ryzen 5 3600XT (6-Core Processor)
    -RAM: 16GB
- 배포 환경:EC2 t3.large (vCPU 2개, RAM 8GB)

3.기술적 선택지
- Pessimistic Lock(비관적 락):경쟁 상황을 방지하기 위해 응모 시점에서 데이터베이스 레벨에서 락을 걸어 동시 접근을 차단.
    -단점: 락을 많이 사용하면 성능 저하 발생 가능.
- Optimistic Lock(낙관적 락):경쟁이 발생하더라도 우선 데이터를 업데이트하고, 충돌 발생 시 재시도(Retry)하는 방식.
    -단점: 재시도 횟수 증가시 성능 저하 가능성.`

1. **배경**
    - 서비스에서 발생한 500번대 예외를 Slack Webhook을 통해 전송하여 모니터링하는 기능을 구현함

    - Slack Webhook API 호출 시, 일부 예외에서만 400 Bad Request 응답과 함께 invalid_payload 반환
    - 로그 확인 결과, 모든 예외는 @GlobalExceptionHandler를 정상적으로 거쳤으므로, 예외 핸들링 과정에서의 문제는 아닌 것으로 판단
    - Slack API 문서에 invalid_payload는 JSON이 잘못 구성되었거나, 메세지 텍스트가 제대로 이스케이프되지 않은 경우 발생한다고 명시됨
    - 발송 실패 메세지 분석 결과, 항상 특수문자(", \)를 포함하고 있었음
    - JSON 이스케이프 처리 문제로 인해 메세지가 유효하지 않은 JSON 형식으로 전송된 것으로 결론

3. **해결 방안**
    - 메세지가 올바르게 파싱 되도록 서비스에 특수문자(", \, \n, \r) 처리 로직 추가
      ![img_6](https://github.com/user-attachments/assets/7528bf34-6bcd-465b-9312-6ee3c448903a)

    - 이를 통해 Slack 메시지 전송 정상화

## 4. 인프라 아키택처 및 적용 기술
### < 아키텍처 다이어그램 >
![image](https://github.com/user-attachments/assets/2f79349e-86d9-42fa-a0a1-1bc68bdce660)


- 위 아키텍처는 **스트리머 이커머스 서비스** 구조를 나타냅니다.  
- AWS EC2 서버의 내부망에서 Redis, mysql은 통신하며,깃헙 액션의 CI/CD를 통해 exluz와 excluz-scheduler에 자동 배포됩니다.
- 사용자는 https://excluz.org 를 통해 요청하고 응답 받는 구조입니다. 

<details>
<summary><b>📦 적용 기술 상세보기</b></summary>

### ***인프라 및 배포***
1. **Github Actions**
    - ***적용 위치***: CI/CD 파이프라인
    - ***사용 이유***: 자동화된 코드 품질 검사와 배포 구현.

2. **AWS ec2 리눅스 서버**
    - **서버 사양**: t3.large : vCPU 2개, RAM 8GB, EBS(gp3): 100GiB, 3000IOPS, 125MB/s
    - **사용 이유**: 제한된 비용 이내 클라우드 기반 배포

---

### ***서버***
1. ***Excluz***
    - **적용 위치**: AWS ec2 리눅스 서버(8080포트, 8081포트)
    - **사용 이유**: 회원, 가게, 물품, 이벤트, 장바구니, 주문, 포인트, 매출 등의 CRUD

2. **Excluz-scheduler**
    - **적용 위치**: AWS ec2 리눅스 서버(9090포트)
    - **사용 이유**: 포인트 거래 내역 기반  매출액/정산/랭킹 스케쥴러

3. **Mysql**
    - **적용 위치**: AWS ec2 리눅스 서버의 docker(3306포트)
    - **사용 이유**: 회원, 가게, 물품, 이벤트, 장바구니, 주문, 포인트, 매출 등 관리

4. **Redis**
    - **적용 위치**: AWS ec2 리눅스 서버의 docker(6379포트)
    - **사용 이유**: 물품, 장바구니, 주문, 랭킹 조회 성능 향상


</details>

## 5. 주요기능



<details>
<summary><b>📦 주요기능 상세보기</b></summary>

### 🌟 **회원가입/로그인: 이메일 인증과 JWT, Spring Security 적용**
- 인증/인가 체계를 Stateless 방식으로 구현하여 확장성 및 유지보수 용이성 향상
- 이멘일 인증을 통한 보안 강화

---

### 🌟 **이벤트 선착순 응모: 비관적/낙관적 락 방식 적용 처리**
- 대용량 트래픽을 수용하기 위해 비관적/낙관적 락을 적용하여 동시성 제어
- 선착순 이벤트 응모 시 이벤트 마감 처리에 따른 당첨자 수 최종 집계 및 검증

---

### 🌟 **주문 처리: Transacional 사용**
- 주문 서비스 관리를 위해 Transacional를 사용
    - **주요 흐름**: `상품 재고확인 & 차감` → `포인트 확인 & 차감` → `포인트 거래내역 생성` → `주문 상태 변경` → `장바구니 제거`
- 오류 발생 시 자동 롤백을 통해 데이터 일관성 유지.
- 향후, sega방식 적용

---

### 🌟 **조회: Redis 캐싱 활용**
- Redis 캐싱을 이용해 많은 사용자가 조회될 것으로 예상되는 데이터에 캐싱 적용
- 상품, 장바구니, 주문, 랭킹 조회에 적용하여 서버 리소스 절약 및 캐싱 효율성 극대화.

---

### 🌟 **결제의 매출액/정산/랭킹: Spring Scheduler 활용**
- 대규모 포인트 결제 데이터를 특정 시간에 자동으로 처리하는 배치 시스템 구축.
    - **프로세스 흐름**: 
      - (매출액) `포인트 거래내역 조회` → `기간별 매출액 집계` → `매출액 결과 저장`
      - (정산) `월 매출액 조회` → `매출액 정산` → `수수료 부과` → `정산 결과 저장`
      - (랭킹) `실시간/일/월/년 메츨액 조회` → `매출액 랭킹 산정` → `랭킹 결과 저장`


</details>
